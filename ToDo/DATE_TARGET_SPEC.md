# Quest Sticky ToDo 日付ターゲット仕様書

## 目的

この仕様書は、Quest Sticky ToDo の「付箋を日付レーンへ置いたときの日付判定」を安定させ、今後の修正で挙動がぶれないようにするための基準を定める。

現在の問題は、日付判定の実装が複数ファイルで上書きされ、判定基準が「ポインタ」「付箋中心」「付箋左端」「付箋全体の重なり」の間で揺れていることにある。これにより、ユーザーの見た目の認識と内部の日付判定がずれやすくなっている。

本仕様では、ユーザーが画面上で「どの日付に置いたように見えるか」を最優先する。

---

## 用語

### 付箋

タスクを表す黄色いカード。既存タスクのドラッグ対象、または新規ブランチ作成時のゴーストカードを指す。

### 日付レーン

日付ごとに分かれた縦方向または横方向の領域。

PC 横表示では、日付は横方向に並ぶ。スマホ縦表示では、日付は縦方向に並ぶ。

### 日付区切り線

日付レーンの境界線。PC では縦線、スマホでは横線。

### 空白地帯

表示されている日付レーンの右側、またはスマホ縦表示では下側にある余白。既存日付の次の日を指定したい場所として扱う。

### 左側の日付 / 上側の日付

PC 横表示では、区切り線の左側にある日付を指す。

スマホ縦表示では、区切り線の上側にある日付を指す。

---

## 基本方針

日付判定は、ポインタ位置ではなく、付箋が置かれて見える位置を優先する。

ただし、単に付箋が線に少し触れただけで区切り線扱いにすると誤判定が増えるため、判定には明確な優先順位を設ける。

---

## 判定基準

### 1. 日付レーン内に置いた場合

付箋が日付レーン内に置かれた場合、その日付を対象日とする。

例:

```text
Jun 28 のレーンに置く → 2026-06-28
Jul 3 のレーンに置く → 2026-07-03
```

既存タスクの場合は、その日付へ直接移動する。

新規ブランチの場合は、作成モーダルの初期日付をその日付にする。

---

### 2. 日付区切り線に置いた場合

付箋を日付区切り線に置いた場合、右側または下側に表示されている日付へ直接飛ばさない。

区切り線は「左側 / 上側の日付の次の日」を指定する場所として扱う。

例:

```text
Jun 28 | Jul 3
```

この区切り線に置いた場合:

```text
2026-06-29
```

を候補日とする。

表示上の次の日付が `Jul 3` でも、区切り線に置く操作の意味は「Jun 28 の次の日に置きたい」である。

---

### 3. 日付レーンの右側 / 下側の空白に置いた場合

付箋をある日付レーンの右側、またはスマホ縦表示では下側の空白に置いた場合、その日付の翌日を候補日とする。

例:

```text
Jun 28 の右側空白 → 2026-06-29
Jun 30 の右側空白 → 2026-07-01
Jul 3 の右側空白 → 2026-07-04
```

これは「次に表示されている日付」ではなく、「直前の日付 + 1日」を採用する。

---

### 4. 最後の日付よりさらに外側の空白に置いた場合

最後に表示されている日付の右側、またはスマホ縦表示では下側に置いた場合、最後の日付の翌日を候補日とする。

例:

```text
最後の日付が Jul 3
その右側に置く → 2026-07-04
```

---

## モーダル表示ルール

### 既存タスクを移動する場合

| 置いた場所 | 挙動 |
|---|---|
| 日付レーン内 | その日付へ直接移動 |
| 日付区切り線 | 日付変更モーダルを表示 |
| 空白地帯 | 日付変更モーダルを表示 |

日付変更モーダルの初期値は、判定された候補日とする。

例:

```text
Jun 28 | Jul 3 の区切り線に置く
→ 日付変更モーダル
→ 初期値 2026-06-29
```

---

### 新規ブランチを作成する場合

| 置いた場所 | 挙動 |
|---|---|
| 日付レーン内 | 作成モーダルを表示し、初期日付はその日付 |
| 日付区切り線 | 作成モーダルを表示し、初期日付は左側 / 上側の日付 + 1日 |
| 空白地帯 | 作成モーダルを表示し、初期日付は直前の日付 + 1日 |

`line` や `blank` 判定になった場合でも、`todayISO()` へ戻してはいけない。

---

## 判定優先順位

日付判定は次の順番で行う。

1. 日付区切り線に置いたか
2. 日付レーン内に置いたか
3. レーン右側 / 下側の空白に置いたか
4. 最後の日付より外側か

ただし、「付箋が少しでも線に重なったら区切り線扱い」は誤判定を生むため禁止する。

区切り線判定は、ユーザーが明確に線へ置いたと感じる範囲に限定する。

---

## 推奨する実装方針

### 1. 日付判定関数を1つに集約する

日付判定は単一の関数で行う。

推奨する返り値:

```js
{
  kind: "lane" | "line" | "blank" | "none",
  date: "YYYY-MM-DD" | null,
  mode: "snap" | "ask" | "free"
}
```

### 2. 呼び出し側は `date` を捨てない

`kind` が `line` や `blank` の場合でも、返された `date` を必ず使う。

禁止例:

```js
const defaultDate = hit.kind === "lane" ? hit.date : todayISO();
```

推奨例:

```js
const defaultDate = hit.date || todayISO();
```

### 3. `kind` は挙動の切り替えだけに使う

`kind` は、直接吸着するか、モーダルを出すかを決めるために使う。

```js
if (hit.mode === "snap") {
  // 直接移動
}

if (hit.mode === "ask") {
  // hit.date を初期値にしてモーダル表示
}
```

## PC 横表示での仕様

PC 横表示では、日付は左から右へ進む。

| 場所 | 候補日 |
|---|---|
| レーン内 | そのレーンの日付 |
| 区切り線 | 左側の日付 + 1日 |
| レーン右側の空白 | 直前の日付 + 1日 |
| 最後の右側 | 最後の日付 + 1日 |

---

## スマホ縦表示での仕様

スマホ縦表示では、日付は上から下へ進む。

| 場所 | 候補日 |
|---|---|
| レーン内 | そのレーンの日付 |
| 区切り線 | 上側の日付 + 1日 |
| レーン下側の空白 | 直前の日付 + 1日 |
| 最後の下側 | 最後の日付 + 1日 |

---

## 折り畳み済み日付との関係

完了済み日付が折り畳まれている場合でも、日付判定の考え方は変えない。

折り畳み済みレーンの右側 / 下側に置いた場合は、その折り畳み日付の翌日を候補日とする。

例:

```text
Jun 29 が折り畳み済み
Jun 29 の右側に置く → 2026-06-30
```

ブランチ線の表示処理と日付判定処理は分離する。

折り畳みブランチ線が正常に見えている場合、日付判定修正でブランチ線を触らない。

---

## 受け入れ条件

以下を満たせば仕様通りとする。

### ケース1: 日付レーン内

```text
Jun 28 レーン内に置く
```

期待結果:

```text
2026-06-28
```

### ケース2: 飛び日付の区切り線

```text
Jun 28 | Jul 3 の区切り線に置く
```

期待結果:

```text
2026-06-29
```

### ケース3: 日付レーン右側の空白

```text
Jun 30 の右側空白に置く
```

期待結果:

```text
2026-07-01
```

### ケース4: 最後の日付の右側

```text
2026-07-04

```

## 実装時に避けること

- 複数ファイルで `hitTestDateArea()` を何度も上書きしない。
- `line` / `blank` の `date` を `todayISO()` で上書きしない。
- ブランチ線の修正と日付判定の修正を同時に混ぜない。
- ポインタ位置だけで日付を決めない。
- 付箋が少しでも区切り線に触れたら `line` 判定、という過剰判定にしない。

---

## デバッグ表示

実装確認中は、以下の情報を画面に表示できるとよい。

```text
kind: lane / line / blank / none
date: YYYY-MM-DD
mode: snap / ask / free
basis: left-edge / center / pointer
```

デバッグ表示は開発用であり、安定後は非表示または開発モード限定にする。

## 今後の整理方針

現在は `ux-fix.js`, `safety-fix.js`, `final-fix.js`, `date-target-fix.js` のように修正レイヤーが積み重なっている。

製品品質を保つため、最終的には以下を目指す。

1. 日付判定処理を1か所に統合する。
2. 折り畳み表示処理と日付判定処理を分離する。
3. 新規作成、既存移動、スマホ操作で同じ日付判定関数を使う。
4. 仕様書の受け入れ条件を満たすことを確認してから、追加のUX改善を行う。
