# Quest Sticky ToDo Layout and Schedule Spec

## 1. この仕様書の目的

この文書は、Quest Sticky ToDo の整列機能・日付なしタスク・日付あり/なし混在時の表示ルールを定義する仕様書である。

`PRODUCT_VISION.md` では、Quest Sticky ToDo の完成形として以下を定義した。

```text
流れで整理する
一覧で実行する
```

この文書では、そのうちボード画面の中核になる以下を扱う。

- 自動整列の欠陥をどう解決するか
- 日付を指定しないタスクをどう扱うか
- 日付・時刻を今後どう拡張するか
- 日付ありタスクと日付なしタスクが混ざった場合に、どう見やすく配置するか

---

## 2. 現在の課題

### 2.1 整列機能の課題

現在の整列機能は、分岐やルートタスクが増えるほど下方向へ伸びやすい。

そのため、タスク数が増えたときに以下の問題が起きる。

- 画面が縦に長くなりすぎる
- ルートタスク同士のまとまりが見えにくくなる
- 分岐が多いタスクほど場所を取りすぎる
- スマホで全体像を確認しにくくなる
- 自動整列後の見た目が、必ずしも分かりやすくない

現在の整列は、構造を単純に縦方向へ逃がす傾向がある。

今後は、ルートタスク単位のまとまりを保ちながら、なるべくコンパクトに配置する必要がある。

### 2.2 日付機能の課題

現在は、タスクに日付がある前提で設計されている。

しかし実際のタスク管理では、必ずしもすべてのタスクに日付があるとは限らない。

例:

```text
数3ワーク
  ├ どこまでやるか決める      日付なし
  ├ 演習12ページ              6/30
  ├ 間違えた問題を復習する    日付なし
  └ 演習13ページ              7/1
```

このように、流れの中には日付があるタスクと日付がないタスクが混ざる。

現在の仕様では、日付を指定しないで自然にタスクを作ることが難しい。

また、日付なしを `today` として扱ってしまうと、今日やるタスクと未定タスクが混ざってしまう。

そのため、日付なしは `今日` ではなく `未定` として扱う必要がある。

### 2.3 日付あり/なし混在の課題

日付ありタスクと日付なしタスクが同じブランチ内に混ざると、単純な日付レーン配置では破綻しやすい。

例:

```text
A  6/29
B  未定
C  未定
D  7/1
```

この場合、B と C は流れとしては A と D の間にある。

しかし、B と C には日付がないため、6/30 や 7/1 のレーンに無理やり置くと、日付があるように誤解される。

つまり、日付なしタスクには、以下の2つを分けて考える必要がある。

```text
実際のスケジュール: 未定
見た目上の配置: 前後の流れに合わせる
```

## 3. 基本方針

Quest Sticky ToDo のボード画面は、日付だけでタスクを並べる画面ではない。

ボード画面の主役は、ルートタスクから枝を伸ばして、やることの流れを作ることである。

日付レーンは、その流れを補助する軸である。

そのため、今後の基本方針を以下のように定義する。

```text
フローが主役
日付は補助軸
```

### 3.1 日付は必須ではない

すべてのタスクに日付を強制しない。

日付がないタスクは、未定タスクとして扱う。

```text
日付なし = 未定
今日 = 今日
```

この2つを混同してはいけない。

### 3.2 日付ありタスクは日付レーンを優先する

日付が指定されているタスクは、日付レーン上の対応する位置へ配置する。

日付ありタスクは、ボード上で期日が視覚的に分かることを優先する。

### 3.3 日付なしタスクはフロー構造を優先する

日付が指定されていないタスクは、日付レーンに無理やり吸着させない。

親子関係・ブランチ構造・前後のタスクとの関係を優先して配置する。

### 3.4 操作中は自由、確定後は整列

ユーザーがドラッグしている最中は、勝手に吸着しない。

確定後に必要な整列を行う。

この方針は、新規タスク作成・既存タスク移動・日付なしタスク作成のすべてで守る。

## 4. スケジュールデータ構造

現在の `targetAt` は日付文字列を前提としている。

しかし、今後は日付なし・日付あり・日時指定を扱う必要がある。

そのため、将来的なデータ構造として `schedule` オブジェクトを導入する。

```js
schedule: {
  type: "none" | "date" | "datetime",
  date: "2026-06-30" | null,
  time: "18:30" | null
}
```

### 4.1 `type: "none"`

日付なし・未定を表す。

```js
schedule: {
  type: "none",
  date: null,
  time: null
}
```

意味:

- 期日は未定
- 今日扱いにはしない
- 日付レーンには固定しない
- フロー構造を優先して配置する

### 4.2 `type: "date"`

日付だけ指定されたタスクを表す。

```js
schedule: {
  type: "date",
  date: "2026-06-30",
  time: null
}
```

意味:

- 指定日のタスク
- 日付レーンに配置する
- リスト画面では期日として表示する

### 4.3 `type: "datetime"`

日付と時刻の両方が指定されたタスクを表す。

```js
schedule: {
  type: "datetime",
  date: "2026-06-30",
  time: "18:30"
}
```

意味:

- 指定日の指定時刻に近いタスク
- 日付レーンでは日付単位で配置する
- リスト画面では時刻も表示する
- 将来的には、日付内で時刻順に並べる

### 4.4 既存データとの互換性

当面は既存の `targetAt` を維持しつつ、内部的に `schedule` へ移行できる形を目指す。

移行方針:

```text
旧: targetAt: "2026-06-30"
新: schedule: { type: "date", date: "2026-06-30", time: null }
```

日付なしを導入する段階で、`targetAt` の空文字や null を `today` に変換する処理は避ける。

## 5. 日付なしタスクの作成仕様

### 5.1 モーダルで日付を空欄にできる

タスク作成・編集モーダルでは、日付欄を空欄にできるようにする。

日付欄が空欄の場合、そのタスクは未定タスクとして保存する。

```text
日付欄が空欄
→ schedule.type = "none"
```

### 5.2 日付レーン内に置いた場合

日付レーン内に明確に置いた場合は、その日付を候補として扱う。

```text
Jun 30 のレーンに置く
→ 初期値: 2026-06-30
```

ただし、モーダルで日付を消せば未定にできる。

### 5.3 区切り線・空白地帯に置いた場合

区切り線や空白地帯に置いた場合は、従来通り候補日を初期値としてモーダルを開く。

例:

```text
Jun 29 | Jun 30 の区切り線
→ 初期値: 2026-06-30
```

日付が飛んでいる場合:

```text
Jun 28 | Jul 3 の区切り線
→ 初期値: 2026-06-29
```

ただし、ユーザーはモーダル上で日付を空欄にできる。

### 5.4 日付レーン外に置いた場合

日付レーンと関係ない場所に置いた場合は、日付なしを初期値として作成できるようにする。

```text
レーン外に置く
→ 初期値: 日付なし
```

これにより、流れだけを作りたい場合に日付を強制しない。

## 6. 整列機能の再設計

### 6.1 現在の整列の問題

現在の自動整列は、分岐やルートタスクが増えるほど下方向へ広がる。

これは単純で実装しやすいが、完成形としては見づらい。

今後は、以下を満たす整列を目指す。

- ルートタスクごとにまとまりを作る
- まとまり同士を見やすく配置する
- ブランチを必要以上に下へ伸ばさない
- 日付ありタスクは日付レーンへ寄せる
- 日付なしタスクはフロー構造を優先する
- スマホでも全体像を追いやすくする

### 6.2 ルート単位のサブツリー

ルートタスクとその子孫を、1つのサブツリーとして扱う。

```text
Root A
  ├ A1
  └ A2

Root B
  ├ B1
  └ B2
```

自動整列では、まず各ルートタスクのサブツリーを個別に整える。

その後、サブツリー同士を画面上に並べる。

### 6.3 サブツリーの配置

サブツリーは、単純に縦へ積み上げるのではなく、必要に応じて横方向・縦方向に分散する。

候補:

```text
PC横表示:
  ルートごとに横方向のまとまりとして配置する
  大きいサブツリーは下へ少し逃がす

スマホ縦表示:
  ルートごとに縦方向のまとまりとして配置する
  ただし、同じルート内では必要以上に縦へ伸ばさない
```

### 6.4 コンパクト整列

分岐が多い場合でも、空白を詰めてコンパクトに配置する。

ただし、線が重なりすぎて構造が読めなくなる場合は、見やすさを優先して余白を取る。

優先順位:

1. 親子関係が読める
2. 日付が読める
3. ルートごとのまとまりが分かる
4. できるだけコンパクト

## 7. 整列モード

今後の整列機能は、1種類だけではなく、複数の考え方を持つ。

### 7.1 フロー整列

親子関係・ブランチ構造を優先する整列。

目的:

- 流れを確認する
- 構造をきれいに見る
- 日付なしタスクも自然に表示する

特徴:

- 日付なしタスクを無理に日付レーンへ乗せない
- 親子関係を優先する
- ルートタスクごとのまとまりを重視する

### 7.2 日付整列

日付レーンを優先する整列。

目的:

- いつやるかを確認する
- 期日順に見る
- 日付ありタスクを明確に並べる

特徴:

- 日付ありタスクをレーンへ寄せる
- 日付なしタスクは未定エリアやフロー上の自然な位置に置く
- 期日が近いタスクを把握しやすい

### 7.3 ハイブリッド整列

完成形での基本整列は、フロー整列と日付整列の中間を目指す。

```text
日付ありタスク: 日付レーンを優先
日付なしタスク: フロー構造を優先
全体: ルートごとのまとまりを維持
```

これを標準の自動整列とする候補にする。

## 8. 日付あり/なし混在時の表示ルール

### 8.1 全部日付あり

すべてのタスクに日付がある場合は、現在のように日付レーンに沿って配置する。

```text
数3ワーク
  ├ 演習12ページ  6/30
  └ 演習13ページ  7/1
```

この場合、日付レーンが主な配置基準になる。

### 8.2 全部日付なし

すべてのタスクに日付がない場合は、日付レーンを基準にしない。

```text
数3ワーク
  ├ ワークを開く
  ├ 演習12ページ
  └ 丸付けする
```

この場合、普通のフロー図としてコンパクトに配置する。

### 8.3 途中から日付あり

途中から日付が出てくる場合、日付なしタスクは日付ありタスクへ向かう準備・前段階として表示する。

```text
数3ワーク
  ├ 準備する        未定
  └ 演習12ページ    6/30
```

表示方針:

- 未定タスクは日付ありタスクの前に置く
- 日付があるようには見せない
- ただし、流れとしてつながっていることは分かるようにする

### 8.4 点々と日付あり

ブランチ内で、日付ありタスクが点々と存在する場合がある。

```text
A  6/29
B  未定
C  未定
D  7/1
```

この場合、B と C は A と D の間の流れとして表示する。

ただし、B と C は未定タスクであることが分かるようにする。

候補表示:

- 未定バッジを付ける
- 日付なしタスクの枠線を少し変える
- 日付ラベルを表示しない
- 日付レーンへ完全には吸着させない

### 8.5 日付なしタスクがルート直下に複数ある場合

ルート直下に日付なしタスクが複数ある場合は、ルートタスクの近くにコンパクトに配置する。

```text
数3ワーク
  ├ 進め方を決める  未定
  ├ 教科書を確認    未定
  └ 演習12ページ    6/30
```

未定タスクは、ルートの作業準備として見える位置に置く。

## 9. 未定タスクの見た目

日付なしタスクは、日付ありタスクと区別できる必要がある。

ただし、強く目立たせすぎるとノイズになる。

候補:

```text
- 期日表示を空にする
- 小さく「未定」バッジを付ける
- 日付レーンの色を使わない
- レーンへの吸着を弱くする
- リスト画面では「期限なし」セクションに入れる
```

未定タスクの見た目は、以下を満たす必要がある。

- 日付がないことが分かる
- しかし、フローの一部として自然に見える
- 日付ありタスクと混ざっても誤解されない

---

## 10. リスト画面との関係

リスト画面では、日付なしタスクを `期限なし` として扱う。

初期状態では、リスト画面の並びは以下とする。

```text
期限切れ
今日
明日
今週
それ以降
期限なし
```

日付なしタスクは、最後の `期限なし` に入れる。

ただし、今日やることとして手動でピン留めするような機能を将来的に追加する場合は、未定タスクでも上に出せるようにする。

## 11. 実装上の注意

### 11.1 `todayISO()` への安易なフォールバックを避ける

日付なしを導入する場合、日付が空のときに `todayISO()` を返す実装は危険である。

```js
// NG
normalizeDate(null) -> todayISO()
```

日付なしと今日が混ざってしまうためである。

今後は、スケジュール用の正規化関数を分ける。

```js
normalizeSchedule(value)
```

のような関数を作り、日付なしを明示的に扱う。

### 11.2 レイアウト用の日付と実データの日付を分ける

日付なしタスクでも、見た目上は前後のタスクに合わせて配置することがある。

そのため、以下を分ける。

```text
schedule: 実際の期日
layoutPosition: 見た目上の配置
```

日付なしタスクに見た目上の位置を与えても、実データとして日付がある扱いにしてはいけない。

### 11.3 デバッグHUD

日付なし対応後は、デバッグHUDにも以下を表示できるとよい。

```text
kind: lane / line / blank / free
scheduleType: none / date / datetime
line: 表示用の日付
candidate: 候補日
target: 保存される日付または none
```

## 12. 受け入れ条件

日付なし・整列改善の受け入れ条件は以下とする。

### 12.1 日付なし作成

- タスク作成モーダルで日付欄を空欄にできる
- 空欄で保存した場合、タスクは未定として保存される
- 未定タスクは今日扱いされない
- リスト画面では期限なしに分類される

### 12.2 日付あり作成

- 日付レーン上で作成した場合、その日付が初期値になる
- 区切り線上で作成した場合、左側/上側の日付 + 1日が初期値になる
- ただし、モーダルで日付を消せば未定にできる

### 12.3 整列

- ルートタスクが増えても、全体が無限に下へ伸びにくい
- ルートごとのまとまりが分かる
- 日付ありタスクは日付レーン上で分かりやすい
- 日付なしタスクは未定であることが分かる
- 日付あり/なしが混ざっても、流れが読める

### 12.4 操作感

- ドラッグ中に勝手に吸着しない
- ドロップ後や保存後に必要な整列が行われる
- ユーザーの意思に反する補正をしない

---

## 13. 開発優先順位

この仕様に関する開発優先順位は以下とする。

1. 日付なしをデータ上で表現できるようにする
2. タスク作成・編集モーダルで日付を空欄にできるようにする
3. 日付なしタスクを今日扱いしないようにする
4. 未定タスクの見た目を決める
5. ルート単位のサブツリー整列を設計する
6. ハイブリッド整列を試作する
7. リスト画面の期限なしセクションへ反映する
8. 将来的に日時指定へ拡張する
