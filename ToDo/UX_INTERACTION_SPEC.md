# Quest Sticky ToDo UX Interaction Spec

## 1. この仕様書の目的

この文書は、Quest Sticky ToDo の細かい操作感・入力UI・削除UI・同日タスクの見せ方を定義する仕様書である。

Quest Sticky ToDo は、ルートタスクから枝を伸ばしてやることの流れを作るアプリである。

そのため、タスクを作る・編集する・削除する操作が重いと、アプリの中心体験である「流れを作る」テンポが悪くなる。

この仕様書では、以下を扱う。

- 全面モーダルを減らす方針
- 作成・編集・削除を操作位置の近くで完結させる仕様
- マウスや指の移動量を減らす仕様
- 同日タスクを単なる縦積みにしない方針
- 日付枠内で親子関係・分岐関係を表示する考え方

---

## 2. 基本方針

Quest Sticky ToDo のUXでは、以下を重視する。

```text
操作している場所の近くで、次の操作が完結すること
```

タスク作成のたびに画面中央へ大きなモーダルを出すと、ユーザーは毎回マウスカーソルや視線を大きく移動する必要がある。

これは、ゲームで毎回遠いメニューを開かされるような面倒さにつながる。

そのため、通常の作成・削除・簡易編集では、全面モーダルではなく、操作対象の近くに小さなポップアップを表示する。

ただし、すべてを小さくしすぎると分かりにくくなるため、以下のように役割を分ける。

```text
軽い操作: 近くのポップアップ
重い操作: 全面モーダルまたは詳細パネル
```

---

## 3. 全面モーダルの課題

現在の作成・日付変更・削除系UIは、画面中央に大きなウィンドウを出す形に近い。

この方式には以下の利点がある。

- 分かりやすい
- 入力欄が大きい
- スマホでも画面中央に集中しやすい

一方で、以下の欠点がある。

- カーソル移動が大きい
- 視線移動が大きい
- ブランチ作成のテンポが止まる
- 連続してタスクを作ると疲れる
- ボード上の文脈から一度切り離される

Quest Sticky ToDo は、枝を伸ばしながら流れを作るアプリなので、作成操作のテンポを特に重視する。

---

## 4. コンテキストポップアップ

### 4.1 定義

コンテキストポップアップとは、操作対象の近くに表示される小さな入力・確認UIである。

例:

```text
ブランチを伸ばして離す
→ その付箋の近くに小さい作成ポップアップが出る
```

```text
削除ボタンを押す
→ 削除対象の付箋の近くに小さい確認ポップアップが出る
```

### 4.2 目的

コンテキストポップアップの目的は、操作量を減らし、流れを作るテンポを保つことである。

- マウス移動量を減らす
- 視線移動を減らす
- 操作していた場所の文脈を維持する
- 連続作成しやすくする
- ボード画面から意識を切り離さない

---

## 5. タスク作成ポップアップ

### 5.1 表示タイミング

新規タスク作成時には、原則として全面モーダルではなく、作成位置の近くに小型ポップアップを出す。

対象:

- ルートタスク作成
- 既存タスクからブランチを伸ばして作成
- 同じブランチへの追加
- 分岐タスクの作成

### 5.2 表示位置

ポップアップは、以下の優先順位で表示位置を決める。

1. 作成予定の付箋の近く
2. ドロップしたカーソル位置の近く
3. 親タスクと新規タスクの間
4. 画面外にはみ出す場合は、画面内に収まるように補正

PCでは、付箋の右下または下側に表示する。

スマホでは、指で隠れにくいように、付箋の少し上または画面下の小さなシートとして表示することも検討する。

### 5.3 入力項目

作成ポップアップの入力項目は、最小限にする。

初期実装では以下のみでよい。

- タスク名
- 期日

ただし、期日は必須にしない。

日付なし仕様と連動して、日付欄は空欄にできる。

将来的には時刻入力にも対応する。

### 5.4 キーボード操作

作成ポップアップでは、以下のキーボード操作をサポートする。

```text
Enter: 作成
Esc: キャンセル
Tab: 入力欄移動
```

タスク名が空欄の状態で Enter を押した場合は、以下のどちらかを検討する。

- 作成しない
- 「新しいタスク」という仮タイトルで作成する

初期仕様では、誤作成を防ぐため、空欄では作成しない方針とする。

### 5.5 外側クリック

ポップアップ外をクリックした場合の挙動は、以下の候補から決める。

候補A:

```text
外側クリック = キャンセル
```

候補B:

```text
タスク名が入力済みなら保存
空欄ならキャンセル
```

Quest Sticky ToDo では軽快さを重視するが、誤保存も避けたい。

初期仕様では、候補Aを採用する。

---

## 6. 日付変更ポップアップ

### 6.1 目的

区切り線や空白地帯に置いたとき、日付変更モーダルを全面表示すると操作が重くなる。

そのため、日付変更も可能であればコンテキストポップアップ化する。

### 6.2 表示内容

表示内容:

- 候補日
- 日付入力欄
- 決定
- キャンセル
- 日付なしにするボタン、または日付欄を空欄にする操作

### 6.3 表示位置

表示位置は、タスクをドロップした場所の近くにする。

既存タスクの移動時は、移動後の付箋付近に表示する。

新規タスク作成時は、作成予定付箋の近くに表示する。

---

## 7. 削除確認ポップアップ

### 7.1 課題

削除は事故が起きると困るため、確認なしにはしない。

一方で、全面モーダルで確認すると操作が重くなる。

### 7.2 方針

削除確認は、削除対象の付箋の近くに小さく表示する。

表示例:

```text
このタスクを削除しますか？
[削除] [キャンセル]
```

### 7.3 子タスクがある場合

削除対象に子タスクがある場合は、通常より強い確認を出す。

表示例:

```text
子タスクも一緒に削除されます
[削除] [キャンセル]
```

この場合は、全面モーダルまたは少し大きめの確認ポップアップを使ってもよい。

---

## 8. 全面モーダルを残す場面

全面モーダルを完全に廃止する必要はない。

以下のような、情報量が多い操作では全面モーダルを残す。

- 詳細編集
- 複数項目の変更
- 子タスクを含む削除確認
- データ削除やリセット
- エクスポート/インポート
- 設定画面

つまり、通常操作は軽く、危険操作や詳細操作は丁寧にする。

---

## 9. 同日タスク表示の課題

現在の仕様では、同じ日付に複数タスクがある場合、単純に縦へ並びやすい。

```text
6/30
  A
  B
  C
  D
```

この表示は、同じ日付にあることは分かるが、タスク同士のつながりが失われる。

Quest Sticky ToDo の主役は、タスクの流れである。

そのため、同じ日付にあるタスクも、単なる縦リストではなく、内部の親子関係・分岐関係を表示する必要がある。

---

## 10. 日付枠内サブフロー

### 10.1 定義

日付枠内サブフローとは、同じ日付に属するタスク群を、その日付の枠内で小さなフロー図として表示する仕組みである。

```text
6/30
┌──────────────┐
│ A ───→ B      │
└──────────────┘
```

単なる縦積みではなく、同じ日付内でも親子関係や分岐を保持する。

### 10.2 目的

目的:

- 同じ日付の中でも作業の流れを失わない
- 親子関係を見えるようにする
- 分岐関係を見えるようにする
- 日付ごとのまとまりを保ちながら、フローも読めるようにする

---

## 11. 日付枠内サブフローの基本例

### 11.1 A から B へつながる場合

すべてのタスクが同じ日付で、A から B へつながっている場合:

```text
A → B
```

日付枠内では以下のように表示する。

```text
6/30
┌──────────────┐
│ A ───→ B      │
└──────────────┘
```

このとき、A と B は同じ日付内の内部フローとして扱う。

### 11.2 A から B, C, D へ分岐する場合

すべてのタスクが同じ日付で、A から複数の子タスクが分岐している場合:

```text
A
├ B
├ C
└ D
```

日付枠内では、分岐として表示する。

```text
6/30
┌──────────────┐
│      ┌→ B     │
│ A ───┼→ C     │
│      └→ D     │
└──────────────┘
```

重要なのは、B, C, D をただ縦に並べるのではなく、A から分岐していることを表示することである。

### 11.3 複数ルートが同じ日付にある場合

同じ日付枠内に複数ルートのタスクがある場合は、ルートごとに小さなグループを作る。

```text
6/30
┌─────────────────┐
│ 数3ワーク        │
│   A ───→ B       │
│                 │
│ 英単語           │
│   C ───→ D       │
└─────────────────┘
```

ルートタスクは、日付枠内でもタグ・グループ名として扱う。

---

## 12. エッジ分類

同日サブフローを実装するには、タスク同士の線を分類する必要がある。

### 12.1 同日エッジ

親タスクと子タスクが同じ日付を持つ場合、その線は同日エッジとして扱う。

```text
A 6/30
└ B 6/30
```

表示:

```text
6/30 枠内の内部線
```

### 12.2 異日エッジ

親タスクと子タスクの日付が異なる場合、その線は異日エッジとして扱う。

```text
A 6/30
└ B 7/1
```

表示:

```text
6/30 枠から 7/1 枠へ伸びる外部線
```

### 12.3 混合エッジ

親タスクと子タスクの片方だけに日付がある場合、混合エッジとして扱う。

```text
A 未定
└ B 6/30
```

または、

```text
A 6/30
└ B 未定
```

表示方針:

- 未定タスク側は日付枠に完全吸着させない
- ただし、流れとしてつながっていることは線で示す
- 日付ありタスク側は日付枠に配置する

### 12.4 未定エッジ

親子ともに日付なしの場合、未定エッジとして扱う。

```text
A 未定
└ B 未定
```

表示:

```text
未定エリアまたはフロー整列内の内部線
```

---

## 13. 日付枠内レイアウトの処理手順

日付枠内サブフローを作るには、以下の処理が必要になる。

```text
1. タスクを日付ごとにグループ化する
2. 同じ日付のタスクだけを取り出す
3. その日付内で parentId / children を見て内部グラフを作る
4. 同日エッジを検出する
5. 同日エッジを日付枠内の線として表示する
6. 分岐がある場合は、枠内で枝分かれとして配置する
7. 異日エッジは日付枠同士をまたぐ線として表示する
8. 未定タスクは未定エリアまたはフロー構造側に配置する
```

この処理により、同じ日付にあるタスクでも、親子関係・分岐関係を失わない。

---

## 14. 日付枠の拡張

同じ日付に多くのタスクがある場合、その日付枠は必要に応じて広がる。

ただし、無制限に広げると見づらくなるため、以下の方針を持つ。

```text
少数タスク:
  コンパクトな枠

分岐が多いタスク:
  枝が読める程度に枠を拡張

大量タスク:
  枠内スクロール、折りたたみ、またはズーム表示を検討
```

初期実装では、枠内スクロールや折りたたみまでは行わず、必要に応じて枠を拡張するだけでよい。

---

## 15. 実装フェーズ

日付枠内サブフローは難易度が高いため、段階的に実装する。

### Phase 1: 同日親子関係の検出

- 同じ日付内の親子関係を検出する
- A → B を同日フローとして認識する
- A → B, C, D のような分岐を認識する
- まだ見た目は簡易的でもよい

### Phase 2: 日付枠内配置

- 同日タスクを日付枠内で配置する
- 親子関係を内部線で表示する
- 分岐を縦積みではなく枝分かれとして表示する

### Phase 3: 異日エッジとの分離

- 同日エッジは枠内線
- 異日エッジは日付枠間の外部線
- 混合エッジは未定タスクと日付枠をつなぐ線

### Phase 4: 未定タスクとの統合

- 未定タスクを日付枠に誤吸着させない
- 未定タスクと日付ありタスクの流れを自然に表示する
- 未定エリアやフロー整列と統合する

### Phase 5: 多数タスクへの対応

- 日付枠の拡張
- 折りたたみ
- ズーム
- 枠内スクロール

---

## 16. 受け入れ条件

### 16.1 コンテキストポップアップ

- 新規タスク作成時、作成位置の近くに入力UIが出る
- マウス移動量が減る
- Enterで作成できる
- Escでキャンセルできる
- 画面外にはみ出さない
- スマホでも入力欄が操作できる

### 16.2 削除確認

- 削除確認が対象付箋の近くに出る
- 子タスクがある場合は、子も削除されることが分かる
- 危険操作では十分な確認がある

### 16.3 同日サブフロー

- A → B が同じ日付の場合、日付枠内でつながりが分かる
- A → B, C, D が同じ日付の場合、分岐として分かる
- 同日タスクをただ縦に並べるだけにしない
- 異なる日付へ伸びる線と、同じ日付内の線を区別できる
- ルートごとのまとまりが維持される
