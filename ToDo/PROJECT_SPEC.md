# Quest Sticky ToDo Project Specification

## 2. プロダクト概要

Quest Sticky ToDo は、タスクをただ並べるのではなく、やることの流れを作れるタスク管理アプリである。

普通のToDoアプリでは、タスクは単独のリストとして並ぶ。

```text
演習12ページ
演習13ページ
英単語
物理プリント
```

Quest Sticky ToDo では、ルートタスクから枝を伸ばし、タスク同士の親子関係・順番・分岐を見えるようにする。

```text
数3ワーク
  ├ 演習12ページ
  └ 演習13ページ
```

中心となる価値は以下である。

```text
やることの流れを作って、今日やることを見失わないToDoアプリ
```

---

## 3. 基本コンセプト

### 3.1 流れで整理する

Quest Sticky ToDo の中心は、タスク同士の関係を視覚化することである。

タスクは単独で存在するだけではなく、あるルートタスクから派生した作業として扱われる。

例:

```text
数3ワーク
  ├ 演習12ページ
  │   └ 丸付け
  └ 演習13ページ
```

この構造により、以下が分かる。

- 何のためのタスクなのか
- どの作業から続いているのか
- どこで分岐しているのか
- 次に何をすればいいのか

### 3.2 一覧で実行する

ボード画面は、構造を確認するには強いが、毎回ボードを見て今日やることを探すのは重い。

そのため、完成形ではリスト画面を用意する。

```text
ボード画面 = 流れを作る・構造を見る場所
リスト画面 = 今日やること・期日が近いタスクを見る場所
```

### 3.3 フローが主役、日付は補助軸

Quest Sticky ToDo は日付管理アプリではなく、フロー型ToDoアプリである。

日付レーンは重要な機能だが、主役ではない。

```text
フローが主役
日付は補助軸
```

そのため、すべてのタスクに日付を強制しない。

## 4. 用語定義

### 4.1 タスク

ユーザーが管理する作業単位。

現在の実装では、タスクは付箋のような見た目で表示される。

主な情報:

- タイトル
- 親タスク
- 期日
- 完了状態
- ブランチ種別
- 表示位置

### 4.2 ルートタスク

親を持たないタスク。

Quest Sticky ToDo におけるルートタスクは、単なる作業項目ではなく、タグ・プロジェクト・大分類のような役割を持つ。

例:

```text
数3ワーク
英単語
物理プリント
文化祭準備
個人開発
```

リスト画面では、ルートタスク自体を実行対象として表示するのではなく、配下タスクのラベルとして扱う。

### 4.3 個別タスク

ルートタスクの配下にある、実際に行う作業。

例:

```text
演習12ページ
英単語50個
プリント101番
READMEを修正する
```

リスト画面では主に個別タスクを表示する。

### 4.4 ブランチ

親タスクから子タスクへ伸びるつながり。

Quest Sticky ToDo では、ブランチにより作業の流れを表す。

### 4.5 同じブランチ

親タスクから直線的に続く主系列のタスク。

例:

```text
A → B → C
```

現在の実装では `branchMode: "same"` で表す。

### 4.6 分岐ブランチ

親タスクから別方向へ分岐するタスク。

例:

```text
A
├ B
└ C
```

現在の実装では `branchMode: "branch"` で表す。

### 4.7 日付レーン

ボード上で期日を視覚化するための軸。

PC横表示では左から右へ、スマホ縦表示では上から下へ日付が進む。

### 4.8 未定タスク

日付が指定されていないタスク。

未定タスクは今日扱いしてはいけない。

```text
日付なし = 未定
今日 = 今日
```

## 5. 画面構成

完成形では、最低限以下の画面を持つ。

```text
1. ボード画面
2. リスト画面
3. スマホ向け簡易画面群
```

### 5.1 ボード画面

現在実装されている中心画面。

目的:

- ルートタスクから枝を伸ばす
- タスク同士の親子関係を見る
- 作業の流れを作る
- 日付レーン上で期日を把握する
- タスクをドラッグして配置・日付変更する

主な機能:

- ルートタスク作成
- タスク作成
- ブランチ作成
- タスク編集
- タスク削除
- 完了切り替え
- 日付レーン表示切り替え
- 自動整列
- Undo
- リセット

### 5.2 リスト画面

今後追加する実行用画面。

目的:

- 今日やることを見る
- 期日が近いタスクを見る
- 期限なしタスクを見る
- 完了操作を素早く行う

表示方針:

```text
ルートタスク名
個別タスク名    期日
```

例:

```text
数3ワーク
演習12ページ    6/30
```

### 5.3 スマホ向け画面

スマホでは、PC版のボードをそのまま縮小しない。

スマホ版では以下を検討する。

- リスト中心モード
- ルート単位フローモード
- ボード簡易モード
- ボトムシートによる入力
- 長押し・スワイプ・専用ハンドルによる操作

詳細は `MOBILE_UX_SPEC.md` を参照する。

## 6. 現在実装済みの主な機能

### 6.1 ローカル保存

タスクデータは `localStorage` に保存される。

現在の保存キー:

```text
quest-sticky-todo-v10
```

過去バージョンのキーからの読み込み互換もある。

### 6.2 ルートタスク作成

ツールバーの `＋ ルート` から作成する。

現在は中央モーダルを開く。

今後は、作成位置近くのポップアップまたはスマホではボトムシートへ移行する予定。

### 6.3 ブランチ作成

各付箋の `+` ハンドルから新しいタスクを伸ばす。

PC横表示では、右へ伸ばすと同じブランチ、上下にずらすと分岐として扱う。

スマホ縦表示では、今後タッチ操作向けに再設計する。

### 6.4 タスク移動

既存タスクをドラッグして移動できる。

日付レーンがONの場合、レーン内に置くとその日付へ吸着する。

区切り線や空白地帯に置いた場合は、日付変更UIを開く。

### 6.5 新規タスク作成中の非吸着

新規タスクをブランチから伸ばしている最中は、ゴースト付箋は日付レーンに勝手に吸着しない。

操作中は自由に動き、確定後に必要な整列を行う。

### 6.6 日付レーン判定

日付レーン判定では、以下を分ける。

```text
date: ホットライン表示用の日付
targetDate: モーダル初期値や保存候補に使う日付
```

区切り線に置いた場合、表示線と候補日が一致しない場合がある。

例:

```text
Jun 28 | Jul 3 の区切り線
表示線: 2026-07-03
候補日: 2026-06-29
```

詳細は `DATE_TARGET_SPEC.md` を参照する。

### 6.7 自動整列

現在は、親子関係と日付をもとに付箋位置を再計算する。

ただし、現在の整列は分岐やルートが増えるほど下方向へ伸びやすい。

今後は、ルート単位サブツリー・日付枠内サブフロー・ハイブリッド整列へ発展させる。

### 6.8 完了切り替え

各付箋の完了ボタンで `todo / done` を切り替える。

### 6.9 削除

各付箋の削除ボタン、または選択状態で削除ボタン/キー操作により削除できる。

現在は、削除時に子タスクを親へ再接続する挙動になっている。

今後は、削除確認ポップアップや子タスクを含む削除確認を整理する。

## 7. 日付仕様

### 7.1 現在の仕様

現在のタスクは `targetAt` に日付文字列を持つ。

```js
targetAt: "2026-06-30"
```

空の値は現在 `today` に寄せられやすい。

これは今後の日付なし対応では問題になる。

### 7.2 今後のスケジュール仕様

今後は `schedule` オブジェクトへ移行する。

```js
schedule: {
  type: "none" | "date" | "datetime",
  date: "2026-06-30" | null,
  time: "18:30" | null
}
```

意味:

```text
type: "none"
  日付なし・未定

type: "date"
  日付だけ指定

type: "datetime"
  日付と時刻を指定
```

この設計により、日付なし・日付あり・将来的な時間指定を扱う。

### 7.3 日付なしの原則

日付なしは今日ではない。

```text
未定タスクを todayISO() に変換しない
```

今後は `normalizeDate()` とスケジュール正規化処理を分離する。

## 8. レイアウト仕様

### 8.1 現在のレイアウト

現在は、日付によって主軸位置を決め、ブランチ構造によってトラックを割り当てる。

PC横表示:

```text
x = 日付
y = トラック
```

スマホ縦表示:

```text
y = 日付
x = トラック
```

### 8.2 今後のレイアウト方針

今後は、以下の3種類を考える。

```text
フロー整列:
  親子関係・ブランチ構造を優先

日付整列:
  日付レーンを優先

ハイブリッド整列:
  日付ありは日付レーン、日付なしはフロー構造を優先
```

標準の自動整列は、ハイブリッド整列を目指す。

### 8.3 同日タスクの扱い

同じ日付に複数タスクがある場合、単純な縦積みにしない。

日付枠内で親子関係・分岐関係を持つ小さなフローとして扱う。

例:

```text
6/30
┌──────────────┐
│ A ───→ B      │
└──────────────┘
```

分岐:

```text
6/30
┌──────────────┐
│      ┌→ B     │
│ A ───┼→ C     │
│      └→ D     │
└──────────────┘
```

詳細は `UX_INTERACTION_SPEC.md` を参照する。

## 9. UX方針

### 9.1 操作中は自由、確定後は整列

ユーザーが直接操作している最中に、アプリが勝手に位置を変えるとストレスになる。

そのため、ドラッグ中は自由に動かせるようにする。

確定後に必要な整列や吸着を行う。

### 9.2 全面モーダルを減らす

現在は作成・編集・日付変更で画面中央のモーダルを使う。

今後は、通常操作では操作対象の近くにコンテキストポップアップを出す。

スマホでは、ボトムシートを使う。

### 9.3 危険操作は丁寧にする

削除やリセットなどの危険操作は、軽くしすぎない。

通常削除は付箋近くの確認ポップアップ、子タスクを含む削除やリセットはより強い確認を行う。

## 10. スマホ版の方針

スマホ版は、PC版をそのまま縮小しない。

```text
PC版の思想を保ちながら、スマホに適したUIへ落とし込む
```

スマホでは、ゲームのように簡単でストレスの少ない操作を目指す。

優先順:

1. 今日やることを見る
2. 完了する
3. 軽く追加する
4. ルート単位で流れを見る
5. 軽い編集・移動をする
6. 複雑なボード編集を行う

詳細は `MOBILE_UX_SPEC.md` を参照する。

## 11. OSS開発の基本方針

### 11.1 仕様を先に書く

大きな機能変更を行う前に、仕様書を更新する。

特に以下は必ず仕様化してから実装する。

- データ構造の変更
- レイアウトアルゴリズムの変更
- 日付なし対応
- スマホ操作の変更
- 削除挙動の変更

### 11.2 既存の動作を壊さない

Quest Sticky ToDo は、ユーザーの操作感に強く依存するアプリである。

そのため、見た目上の小さな変更でも操作感が大きく変わる可能性がある。

変更時は、以下を確認する。

- PC横表示
- スマホ縦表示
- 既存タスク移動
- 新規ブランチ作成
- 日付レーン判定
- 区切り線判定
- Undo
- localStorage互換

### 11.3 修正レイヤーを増やしすぎない

現在は、`app.js` の後に複数の fix ファイルを読み込んでいる。

短期的には動作安定のために有効だが、長期的には保守性が落ちる。

今後は安定した修正を `app.js` 本体、または整理されたモジュールへ統合する。

## 12. 実装優先順位

現時点での大きな開発優先順位は以下である。

1. OSS向けドキュメント整備
2. コード構造の整理
3. 日付なしタスク対応
4. スケジュール構造への移行準備
5. コンテキストポップアップ実装
6. リスト画面の最小実装
7. 自動整列の再設計
8. 同日タスクのサブフロー表示
9. スマホ版UI/UX改善
10. fixファイルの本体統合

## 13. 受け入れ条件

Quest Sticky ToDo の基本仕様として、以下を満たすことを目指す。

- ルートタスクから枝を伸ばしてタスクを作れる
- タスク同士の親子関係が見える
- 同じブランチと分岐ブランチを区別できる
- 日付レーン上で期日が分かる
- 日付なしタスクを未定として扱える
- ボード画面で流れを整理できる
- リスト画面で今日やることを確認できる
- スマホではPC版を縮小せず、スマホ向け操作で扱える
- 仕様を知らない開発者でも、ドキュメントから実装方針を理解できる

---

## 14. 今後分離・整理したい領域

現在は単一アプリとして動いているが、今後は以下のように責務を分けると保守しやすい。

```text
data:
  タスクデータ、保存、移行

schedule:
  日付なし、日付、日時

layout:
  自動整列、トラック計算、日付枠内サブフロー

interaction:
  ドラッグ、ブランチ作成、選択、削除

view:
  ボード描画、リスト描画、スマホ表示

ui:
  モーダル、ポップアップ、ボトムシート
```

この分離方針の詳細は `TECHNICAL_ARCHITECTURE.md` を参照する。
